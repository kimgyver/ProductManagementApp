# ProductManagementApp 상세 분석 문서

작성일: 2026-03-11  
분석 기준: 현재 저장소 코드 상태 (`API`, `BackgroundProcessor`, `API.Test`)

---

## 1) 프로젝트의 핵심 목적 (당신의 원래 의도 기준)

이 프로젝트의 가장 중요한 목적은 **실무형 .NET 백엔드 역량을 잊지 않고 유지/강화**하는 데 있습니다.

즉, 단순 CRUD 데모가 아니라 아래를 한 번에 반복 학습하는 용도로 아주 적합합니다.

- Web API 설계와 계층 분리
- 인증/인가(JWT + Session/Cookie)
- 비동기 메시징 기반 아키텍처(SQS)
- 외부 서비스 연동(SES)
- Worker Service 기반 백그라운드 처리
- EF Core + 마이그레이션
- 예외 처리 및 테스트

한 문장으로 정리하면:

> “백엔드 실무에서 자주 쓰는 기술 조합을 작게라도 끝까지 연결해 보는, 취업 유지형 실전 트레이닝 프로젝트”

## 2) 현재 시스템이 실제로 하는 일

### 2.1 도메인 기능

- `User` 관리: 회원 등록/조회/수정/삭제, 로그인/로그아웃
- `Product` 관리: CRUD (관리자 권한 필요)
- 회원가입 후 환영 이메일 비동기 발송
- 이메일 발송 실패 시 사용자 `Verified=false` 업데이트

### 2.2 전체 플로우 (요약)

1. 사용자 회원가입 요청 (`API`)
2. 사용자 DB 저장 (SQLite, EF Core)
3. 이메일 발송 메시지를 SQS로 전송
4. `EmailBackgroundWorker`가 SQS에서 메시지 수신 후 SES 발송
5. 실패 시 `EmailFailureQueue`로 실패 이벤트 전송
6. `EmailFailureBackgroundWorker`가 실패 큐 메시지 수신
7. Worker가 API에 인증 후 `verification-failed` 엔드포인트 호출
8. 해당 사용자 `Verified=false` 반영

이 흐름은 실무에서 자주 나오는 **API + Queue + Worker + 외부 서비스** 구성을 축소판으로 잘 담고 있습니다.

## 3) 아키텍처/코드 구조 분석

## 3.1 솔루션 구성

- `API` (`net8.0`): ASP.NET Core Web API
- `BackgroundProcessor` (`net8.0`): Hosted Worker 2개 실행
- `API.Test` (`net8.0`): xUnit + Moq 기반 단위 테스트

### 3.2 API 레이어 구성

- `Controllers`: 입출력 및 HTTP 책임
- `Services`: 비즈니스 로직 (Command/Query 분리)
- `Infrastructure`: Repository + DbContext
- `Middleware`: 전역 예외 처리
- `DTOs`: 요청 모델/유효성 검증
- `Models`: 엔티티

### 3.3 CQRS 적용 수준

완전 분리형(별도 저장소/버스)까지는 아니지만, 학습 목적에 적절한 **서비스 레벨 CQRS 분리**가 적용되어 있습니다.

- Command: `UserCommandService`, `ProductCommandService`
- Query: `UserQueryService`, `ProductQueryService`

이 방식은 면접/실무에서 “왜 CQRS를 썼는지”를 설명하기 좋은 중간 단계 구현입니다.

## 4) 사용 기술 스택 (상세)

## 4.1 런타임/프레임워크

- .NET 8
- ASP.NET Core Web API
- .NET Worker Service (`BackgroundService`)

### 4.2 데이터/ORM

- EF Core (SQLite Provider)
- 마이그레이션 적용 (`Migrations` 폴더 존재)
- Fluent API:
  - `User.Email` 유니크 인덱스
  - 문자열 길이/필수값 설정
  - `Product.Price` 타입 지정

### 4.3 인증/보안

- JWT Bearer 인증
- Cookie 인증 + Session 동시 사용
- 역할 기반 권한 (`Admin`, `User`)
- 비밀번호 해싱(`PasswordHasher` 래핑 서비스)
- 커스텀 유효성(`NoSpecialCharactersAttribute`)

### 4.4 메시징/외부 연동

- AWS SQS: `SendEmailQueue`, `EmailFailureQueue`
- AWS SES: 이메일 발송
- Worker ↔ API 간 HttpClient 통신

### 4.5 문서화/개발 편의

- Swagger/Swashbuckle
- 글로벌 예외 미들웨어

### 4.6 테스트

- xUnit
- Moq
- Coverlet collector

## 5) 잘한 점 (강점)

- **실무형 흐름을 끝까지 연결**: 등록→큐→워커→외부 서비스→실패보정
- **계층 분리**가 명확하여 리팩터링/확장 기반이 좋음
- **예외 처리 미들웨어**를 도입해 API 응답 일관성 확보
- **Repository + EF Core + Migration** 조합으로 기본기 충실
- 테스트 프로젝트를 이미 분리해 둬서 확장하기 좋음

## 6) 현재 코드 기준 개선 포인트 (실무 감각 관점)

## 6.1 기능/정확성

- `UsersController.Index()`에서 `await` 누락 가능성 (현재 `Task`가 바로 반환되는 형태)
- `UserQueryService.AuthenticateUserAsync()`가 전체 사용자 조회 후 필터링 → 이메일 조건 단건 조회로 개선 필요
- `SessionService.GenerateSessionAsync()` 비동기 호출이 `await` 없이 사용되는 지점 존재

### 6.2 보안/설정

- 설정 파일에 큐 URL/소스 메일/시크릿 형태가 직접 노출되는 구조
- Worker의 client 인증에서 `JWT:Secret`/`Jwt:Secret` 키 사용이 혼재됨
- 로그인 응답에 사용자 엔티티 전체를 노출하는 방식은 축소 DTO 권장

### 6.3 아키텍처/운영

- API와 Worker 모두 재시도/백오프/데드레터 큐 전략이 제한적
- 장애 추적을 위한 Correlation ID/구조적 로깅 고도화 필요
- 테스트가 컨트롤러 일부에 집중되어 서비스/리포지토리/통합 테스트가 부족

### 6.4 코드 품질

- 파일명 오타(`UserQeuryService`) 등 유지보수성 저하 요소
- 사용하지 않는 using/주석 코드 정리 필요

## 7) “취업 유지” 목적에 맞춘 발전 전략

이 프로젝트는 “크게 키우는 것”보다 **학습 루프를 반복하기 좋은 구조**로 발전시키는 것이 우선입니다.

## 7.1 1단계 (1~2주): 기본기 고도화

목표: 코드 신뢰성과 설명력을 올려, 면접에서 “왜 이렇게 설계했는지”를 말할 수 있게 하기

- 입력/출력 DTO 분리 강화 (엔티티 직접 노출 축소)
- 로그인/등록 흐름 유닛 테스트 추가
- `Users`, `Products` 컨트롤러의 정상/예외/권한 케이스 테스트 보강
- 설정값을 User Secrets/환경변수 기반으로 이동
- 비동기 호출(`await`) 누락 및 네이밍 오타 정리

### 7.2 2단계 (2~4주): 실무형 운영 품질

목표: “동작”을 넘어서 “운영 가능한 백엔드” 경험 확보

- Serilog(또는 구조적 로깅) + 요청 추적 ID 추가
- 워커 재시도 정책(예: Polly) + 실패 메시지 처리 정책 명확화
- 헬스체크/레디니스 엔드포인트 도입
- OpenAPI 설명/예시 강화

### 7.3 3단계 (4주+): 포트폴리오 확장

목표: 이력서/깃허브에서 임팩트 있는 포인트 만들기

- SQLite → PostgreSQL 전환 시나리오
- Docker Compose로 API + Worker + LocalStack(or 실제 AWS 분리 환경) 구성
- 간단한 프론트(관리자 상품 관리/사용자 가입) 연동
- CI 파이프라인(빌드/테스트) 추가

## 9) 이 프로젝트를 이력서/면접에서 말하는 방법

### 한 줄 요약

“ASP.NET Core API와 Worker Service를 분리하고 SQS/SES를 연동해, 회원가입 후 이메일 비동기 처리 및 실패 보정까지 구현한 실무형 백엔드 프로젝트”

### 강조 포인트

- 단순 CRUD가 아니라 **비동기 메시징 + 장애 보정 플로우**까지 구현
- 인증(JWT/Session), 예외 처리, 테스트 등 백엔드 필수 요소를 포함
- 계층 분리와 CQRS 스타일로 유지보수/확장성을 고려
